# 晓晓 AI 智能客服产品需求文档

版本：v1.0  
日期：2026-06-26  
项目定位：面向电商售后场景的可上线型 AI 智能客服产品

## 1. 产品概述

晓晓 AI 智能客服是一款面向电商售后履约场景的 Agent 型智能客服系统。产品目标不是做单点聊天机器人，而是让 AI 在真实业务链路中完成“识别问题、补齐信息、调用工具、检索知识、生成回复、必要时转人工”的闭环。

项目当前以七匹狼相关售后场景作为演示业务域，覆盖订单查询、申请退款、修改地址、催发货、面料洗护、尺码选择、退换货规则、高危拦截和人工兜底等核心流程。

## 2. 产品目标

### 2.1 业务目标

- 降低重复性售后咨询的人力处理成本。
- 提升查物流、退款、改地址、催发货等高频问题的处理效率。
- 控制 AI 幻觉，避免错误承诺包邮、赔偿、全额退款、活动权益等高风险内容。
- 在高危、低置信度、知识库无依据场景中稳定转人工。
- 为后续真实上线、二维码演示、简历作品集展示提供可体验产品。

### 2.2 用户体验目标

- 用户进入页面后只看到智能客服聊天面板，不暴露内部运营后台。
- AI 回复需要有明确进度状态，例如正在识别意图、正在查询订单、正在检索知识库。
- 对业务执行结果使用结构化卡片展示，降低用户理解成本。
- 对缺失订单号、退款原因、新地址等必要信息进行反问，而不是直接失败。
- 对无法确认或高风险的问题提供人工客服通道。

### 2.3 产品能力目标

- 支持流式输出，缩短用户等待感。
- 支持基于后端卡片标识动态渲染物流、退款、改地址、催办、知识库依据、转人工等卡片。
- 支持 Agent 主控工作流，而不是简单问答。
- 支持 RAG 检索增强生成与防幻觉约束。
- 支持后台运营监控、链路追踪、知识库治理、人工坐席处理。

## 3. 用户与角色

| 角色 | 目标 | 可访问页面 | 核心能力 |
| --- | --- | --- | --- |
| 消费者 | 快速解决售后问题 | `/` 用户端聊天页 | 咨询、查单、退款、改地址、催发货、反馈 |
| 人工坐席 | 接管 AI 无法处理的问题 | `/agent` 坐席工作台 | 查看工单、理解转人工原因、完成处理 |
| 运营管理员 | 监控系统质量并治理知识库 | `/admin` 管理后台 | 查看指标、模型健康、链路追踪、知识库召回 |
| 面试官或体验者 | 扫码体验产品能力 | `/` 用户端聊天页 | 只体验真实用户视角 |

## 4. 使用场景

### 4.1 订单查询

用户说“查物流 202606160001”或“我的订单到哪里了”，系统识别为查询/操作类意图。如果订单号齐备，调用 QueryOrderStatus 工具并返回物流状态卡片；如果订单号缺失，返回订单选择卡片。

### 4.2 申请退款

用户说“我要退款 202606160001，原因是尺码不合适”，系统提取订单号和退款原因，调用 ApplyRefund 工具，返回退款受理结果和预计到账时间。若缺少退款原因，系统继续追问。

### 4.3 修改地址

用户说“修改地址 202606160002，新地址是上海市徐汇区某某路 88 号”，系统提取订单号和新地址，调用 UpdateAddress 工具。若订单已进入运输或签收阶段，系统提示已打开人工拦截通道。

### 4.4 催发货

用户说“帮我催一下订单 202606160002”，系统调用 UrgeLogistics 工具并返回催办回执。

### 4.5 知识咨询

用户询问“七匹狼衬衫怎么洗”“尺码怎么选”“退换货规则是什么”等问题，系统进入 RAG 检索流程。只有知识库召回内容包含依据时才组织回复；没有依据时按防幻觉兜底话术转人工。

### 4.6 高危或敏感内容

当用户输入涉及隐私泄露、诈骗、威胁、自伤、监管投诉等高风险表达时，系统不继续普通对话，直接生成转人工卡片并记录审计日志。

## 5. 产品范围

### 5.1 当前版本范围

- React 用户端聊天页面。
- FastAPI 后端服务。
- 流式聊天接口 `/chat/stream`。
- 会话初始化接口。
- 四个 Mock 业务执行工具。
- Few-shot 意图分类器和关键词兜底。
- 槽位填充与订单选择卡片。
- RAG 知识库问答与防幻觉兜底。
- 高危拦截、低置信度澄清、连续失败转人工。
- 人工坐席工作台。
- 运营管理后台。
- 合规偏好设置与负反馈入口。
- UAT 自动化评测与报告。

### 5.2 当前版本不包含

- 真实电商订单系统对接。
- 真实支付退款链路。
- 企业微信、淘宝、抖音、京东等平台客服通道对接。
- 多租户权限体系。
- 真实客服排班和 SLA 分配。
- 生产级日志平台、监控平台和告警平台。

## 6. 功能需求

### 6.1 用户端聊天

| 功能 | 说明 | 优先级 |
| --- | --- | --- |
| 消息输入 | 用户可以输入自然语言问题 | P0 |
| 快捷操作 | 提供查物流、退款、改地址、尺码咨询等快捷入口 | P0 |
| 流式回复 | 后端以 NDJSON 流形式返回状态、文本和卡片 | P0 |
| 结构化卡片 | 按卡片类型渲染订单、退款、改地址、催办、RAG、转人工 | P0 |
| 订单选择 | 缺订单号时展示七匹狼相关订单选择卡片 | P0 |
| 合规入口 | 展示个性化标签拒绝权和负反馈入口 | P0 |

### 6.2 Agent 主控流程

| 节点 | 说明 | 验收标准 |
| --- | --- | --- |
| 进线预处理 | 初始化用户和近 30 天订单上下文 | 会话开始可获取订单摘要 |
| 意图路由 | 判断查询/操作、知识咨询、高危兜底 | 分类准确率达到 UAT 标准 |
| 槽位填充 | 判断订单号、退款原因、新地址是否齐备 | 缺槽位时返回反问卡片 |
| 工具调用 | 调用后端业务工具获取结构化结果 | 四个核心工具可返回 JSON |
| 生成与监控 | 生成自然语言和卡片，记录 trace | 后台可查看调用链路 |

### 6.3 意图分类与异常降级

系统需要将用户输入分为三类：

| 分类 | 处理流程 |
| --- | --- |
| 查询/操作类 | 进入槽位填充和工具调用 |
| 知识/咨询类 | 进入 RAG 检索和防幻觉生成 |
| 闲聊/高危类 | 触发拦截或转人工 |

异常降级要求：

- 当分类置信度低于 0.85 时，暂停业务执行，返回澄清引导和快捷卡片。
- 当 LLM 分类超时超过 2000ms 时，降级为关键词和规则匹配。
- 同一会话连续 3 次低置信度，强制转人工。

### 6.4 RAG 与防幻觉

知识咨询类问题必须进入知识库检索流程。

RAG 要求：

- 知识库按语义段落或 QA 对组织。
- 每条知识绑定 metadata，例如品牌、类目、场景、生效时间等。
- 检索时结合向量召回和关键词匹配。
- 生成回答时只能基于召回内容。

防幻觉兜底话术必须保持一致：

> 抱歉，目前系统暂未查阅到该商品的具体活动规则，我马上为您转接人工客服核实。

### 6.5 人工兜底

以下场景必须转人工：

- 高危、敏感、隐私、监管投诉等风险表达。
- 连续 3 次低置信度识别失败。
- 知识库未召回足够依据。
- 订单状态导致自动工具无法完成，例如运输中订单改地址。

转人工卡片需要包含原因和工单编号，后台需要保留审计日志。

### 6.6 运营后台

运营后台面向内部使用，不能作为公开用户入口。

核心能力：

- 查看系统运行状态。
- 查看知识库召回和治理信息。
- 查看模型健康状态。
- 查看链路追踪记录。
- 查看负反馈、合规偏好和转人工情况。

### 6.7 坐席工作台

坐席端用于承接 AI 转人工工单。

核心能力：

- 查看待处理工单。
- 查看转人工原因和用户原始问题。
- 标记工单已处理。
- 记录处理备注。

## 7. 页面与信息架构

| 页面 | 路径 | 访问对象 | 说明 |
| --- | --- | --- | --- |
| 用户端聊天页 | `/` | 消费者、面试官 | 公开扫码入口，只展示客服体验 |
| 人工坐席工作台 | `/agent` | 内部坐席 | 处理转人工工单 |
| 运营管理后台 | `/admin` | 内部运营 | 管理知识库、查看指标和追踪 |

公网部署时，`/agent` 和 `/admin` 必须启用内部令牌，避免用户看到内部运营信息。

## 9. 数据与埋点需求

需要记录以下数据，用于评测和后续运营分析：

| 数据类型 | 说明 |
| --- | --- |
| 会话数据 | session_id、用户消息、时间戳 |
| 路由数据 | 意图分类、动作名称、置信度、来源 |
| 工具数据 | 调用工具、参数、返回结果、耗时 |
| RAG 数据 | 召回分片、metadata、得分、是否命中 |
| 转人工数据 | 工单编号、转人工原因、用户原话、处理状态 |
| 合规数据 | 个性化拒绝权、负反馈原因、反馈内容 |
| 性能数据 | TTFT、总耗时、接口错误 |

## 10. 验收指标

| 指标 | 目标 | 当前验收方式 |
| --- | --- | --- |
| TTFT | < 800ms | evaluator.py 自动测试 |
| 意图分类准确率 | >= 92% | 自动测试集 |
| 高危拦截率 | 100% | 高危样本测试 |
| 幻觉率 | < 0.5% | 知识库无依据样本检查 |
| 用户端隔离 | 用户只能看到聊天页 | 页面路由检查 |
| 内部后台保护 | `/admin`、`/agent` 支持令牌保护 | 环境变量配置检查 |

## 11. 合规要求

- 用户端必须展示个性化标签拒绝权。
- 用户端必须提供负反馈入口。
- 不得向普通用户展示后台 trace、模型健康、知识库治理、人工坐席工单等内部信息。
- 敏感内容、高危表达必须优先转人工。
- AI 不得基于未召回依据自主承诺补偿、包邮、全额退款或活动权益。

## 12. 上线与演示要求

### 12.1 简历二维码入口

二维码应指向公网用户端聊天页 `/`，不应直接指向 `/admin` 或 `/agent`。

### 12.2 面试现场演示路径

1. 面试官扫码体验用户端客服。
2. 演示查物流、退款、知识库问答、防幻觉兜底、高危转人工。
3. 自己电脑打开后台，解释 Agent 路由、RAG、trace 和人工接管。
4. 展示 UAT 报告和指标字典，证明产品可评测、可治理。

## 14. 风险与对策

| 风险 | 表现 | 对策 |
| --- | --- | --- |
| 知识库覆盖不足 | 用户咨询被频繁转人工 | 建立未命中问题池，定期补充知识 |
| 模型自由发挥 | 错误承诺权益或赔偿 | 强制 RAG 上下文约束和兜底话术 |
| 后台误公开 | 用户看到内部数据 | 用户端和内部端分路由，生产开启令牌 |
| 响应慢 | 用户等待感强 | 流式状态先返回，分类超时降级 |
| 高危漏拦 | 敏感内容进入普通回复 | 高危词和规则前置，UAT 保持 100% 拦截 |
